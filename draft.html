A Proposal for Extending MIME Types

Abstract

[RFC2046] defines MIME (Multipurpose Internet Mail Extensions) to enable the inclusion of multimedia in mail. This was later extended to HTTP with [RFC6838], and new protocols in development such as IPFS are creating their own binary encodings of similar concepts with "multicodecs". This memo proposes a dramatic expansion of the scope of MIME types for the purpose of universal identification of data types in current and future protocols.¶

1. Introduction

1.1. Reason

Currently, MIME types offer a useful and relatively simple method of identifying multimedia typing in communications over the WWW. However their scope is limited and the relatively involved registration process discourages the development of new, innovative formats such as YAML which has yet to receive a MIME type despite being in wide use even on the web. MIME types are also confused by legacy decisions which cannot be corrected without breaking much of the web, such as a limited number of standards trees, confusion of encoding and content leading to ambiguous distinctions like text/javascript vs application/javascript, and a dramatic overloading of application/* as a catch-all category (a casual check as of the time of writing shows 447 non-vendor application types and 955 vendor types, compare with the next largest tree audio/* with 155 vendor and non-vendor types combined). Additionally, MIME has growing use as file type metadata, for instance in [shared-mime-info]. Evidently, MIME has grown far beyond the original expectation of web media metadata.¶

2. Protocol

2.1. Text

A textual representation is provided to aid in human recognition and backwards compatibility with the current MIME standard.¶

Type and subtype will be defined later but are supersets of those provided by MIME. Suffix is part of subtype, reserved for the same semantics as MIME: the subtype is a subset of the provided format. Parameters are included for compatibility with MIME and normalization encodes it in a different part of the type specifier (eg charset parameter becomes encoding). They may be used to build the extended binary form, but no subtype-independent mapping is possible.¶

"Encoding" is taken to be one of:¶

  1. IANACharset identifiers or their aliases. [RFC3808]¶
  2. "text" to mean an unspecified text encoding (each format suggests a default)¶
  3. "binary" to mean an unencoded octet stream¶
  4. "other" to mean an encoding not yet added to IANACharset¶

  5. A textual binary encoding (ASCII assumed):¶

    1. "percent" [RFC3986]¶
    2. "base32", "base32hex", or "base64" [RFC4648]¶
    3. "hex"¶
    4. "qp" (quoted-printable) [RFC2045]¶
    5. (more can be registered later)¶

2.2. Binary 

[..encoding..|type.........subtype]
|-----10-----|--------22----------|
¶

The binary format is the canonical form - textual representations are inspired by it rather than the other way around. The form intended for long-term use is a 32-bit integer, the ten MSBs allocated for the type and the LSBs allocated for the subtype. Between these allocations is padding made of unset bits. This allows the types and subtypes to be expanded independently - new standards can make use of the unused bits, which older implementations will recognize as unknown (sub)types.¶

Encoding is an coded map to the MIBenum values:¶

  • 0¶

    • 0: 7-8 bits (MIBenum 0-999)¶

      • 0 = "binary"¶
      • 1 = "other"¶
      • 2 = "text" ("unknown" in MIBenum)¶

    • 1¶

      • 0: 5-7 bits (MIBenum 1000-1999)¶
      • 1: 7 bits Text to binary encoding¶
  • 1: 9 bits (MIBenum 2000-2999)¶

Type codes are allocated as a prefix tree of varying base-2 radixes. This allows general information to be determined from category prefixes, and for subtype spaces to be allocated more space based on expected usage.¶

Contypes can optionally include an additional 32-bit integer with per-subtype allocation which encodes other variations of data formatting, such as the codec of an audio container file. An integer value of 0 is reserved to mean no further information is known. In general this value can be safely removed as it encodes metadata found in the file itself. This maps roughly to MIME parameters, though some standard parameters (such as charset) are encoded in other ways.¶

3. Types

Types prefixed with @ are taken to be semantic distinctions which are disallowed from being used as a type in the textual format. Otherwise, any type in the path to a leaf node can be used in the textual format, but the leaf node is to be taken as the canonical form. In general, types are informational, not disambiguating. All subtypes have unique identifiers and aliases. So for instance, media/png and image/png can both be used, though image/png should be taken as the canonical form.¶

What follows is a first draft of the type bit allocation:¶

  • 0 primitive¶

    • 0 number¶

      • 0 int¶
      • 0 real¶

    • 1¶

      • 0 string¶
      • 1 struct¶

  • 1 composite¶

    • 00 data¶

      • 00 config¶

        • 0 @xml¶
        • 1 @json¶

      • 01 crypto¶

        • 0 digest¶

          • 0 hash¶
          • 1 crc¶
        • 1 encryption¶

      • 10 technical¶

        • 0 science¶

          • 0 chemical¶
      • 11 app¶

    • 01 media¶

      • 00 multi¶

        • 0 video¶
        • 1 doc¶

      • 01 audio¶

        • 0 codec¶
        • 1 container¶

      • 10 graphic¶

        • 0 raster¶
        • 1 vector¶

      • 11 other¶

        • 0 style¶

          • 0 font¶
          • 1 sheet¶
        • 1 model¶

    • 10 exec¶

      • 0 compile¶

        • 0 code¶
        • 1 source¶

      • 1 interpret¶

        • 0 script¶
        • 1 dsl¶

    • 11 meta¶

      • 00 proto¶
      • 01 archive¶

      • 10 file¶

        • 0 node¶
        • 1 fs¶

      • 11 misc¶

        • 0¶

          • 0 special¶
          • 1 unknown¶
        • 1 multipart¶

For compatibility with MIME, some additional categories are defined as alias categories with no simple encoding:¶

  • application - split across app and exec¶
  • text - resolves for any subtypes with a text encoding. Anything of the form binary:text/* is an error¶
  • message - mixture of proto and config¶

For further compatibility, additional name rewriting rules are defined:¶

  • example/* = special/example¶
  • */example = special/example¶
  • prs.* = special/personal¶
  • Remove x.* and x-* prefixes. If unresolved, special/nonstandard¶
  • Remove vnd.* prefix and search. If unresolved, special/vendor¶
  • For unresolved types, special/unknown¶
  • For unresolved subtypes, unknown/(type)¶

The type allocated for unstructured data is unknown/blob, aliased as application/octet-stream.¶

4. Registry attributes

Every subtype entry in the registry has associated attributes. These are:¶

  • encoding - the default encoding to assume if none is provided.¶
  • parent - the standard used by this subtype (if any) eg IFF, RIFF, XML, JSON¶

  • tags - a list of freeform tags for keeping track of shared features.¶

    • FourCC - magic follows the FourCC standard.¶
    • experimental - the format specification is not stable and may change over time.¶
    • deprecated - conforming programs SHOULD NOT generate new files of this format, but MAY support reading them.¶
    • obsolete - conforming programs MAY NOT generate new files of this format, MAY support reading them, and if possible, SHOULD attempt to convert to a more modern format. Any format tagged with "obsolete" is guaranteed to have at least one objectively superior format to translate to (capable of holding all information from the obsolete format).¶
    • legacy - an otherwise obsolete format kept for compatibility with legacy hardware/software.¶
    • proprietary - file format standard was created and/or currently owned/maintained by a private organization.¶
    • protected - file format is legally protected and implementing a parser may have legal implications.¶
    • unknown - format is known to exist, but the internal structure is unknown.¶
    • lossy - converting to this format from another with similar usage will lose information¶
    • lossless - no information will be lost in converting to this format¶
    • nonstandard - there is no recognized standardization authority and the existence and usage of the format is known only through defacto usage.¶
    • unbroken-crypto - some or all of the file's structure uses unknown cryptography, often for DRM purposes.¶
    • alias:* - used for alias classes such as application and text.¶
  • magic - identifying bytes (if applicable).¶
  • extensions - list of acceptable file extensions (if applicable) in order of preference.¶
  • standards - a key:value map to convert to other type specifier standards (eg MIME, OSType, UTI).¶
  • names - list of acceptable names for this subtype in order of preference (all of which may be used in the textual form).¶
  • related - co-standards which a format may be converted to without losing information.¶
  • format - specification for the format described for introspection purposes. This may take the form of a natural language specification, but is intended to eventually have a computer-readable form. One recommended for expressing this field is KaiTai Struct, a YAML-based file parsing template language with targets for many languages.¶
